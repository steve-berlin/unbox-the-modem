Part 5 — How Embedded Linux Boots

Turning power on, step by step, until the device is ready.

TL;DR

When you power the device on, it does not wake up all at once. It goes through a fixed set of stages, one after another.

A tiny bit of built-in start code runs first, then a small program called the bootloader, then the Linux operating system, then the first program, then the background programs, and finally the network comes up.

Using UART from Part 4, you can watch every stage scroll past as plain text.

The boot stages, in order

"Booting" just means starting up. Each stage hands over to the next, a bit like a relay race. Here is the whole run:

1. Power on

Electricity reaches the main chip. Nothing clever happens yet — the chip simply comes alive and looks for its very first instruction.

2. Tiny built-in start code

The main chip has a very small amount of start code baked into it at the factory. This code cannot be changed. Its only job is to find and start the next stage — the bootloader — from permanent storage.

3. The bootloader

The bootloader is a small program whose job is to find the operating system in permanent storage, copy it into fast memory (RAM), and start it. Many embedded devices use a bootloader called U-Boot. A bootloader can often also update the operating system, recover a broken one, or run tests.

4. The Linux operating system starts

Now the main software takes over. Linux checks the hardware it can see, switches on the parts it needs, and prepares to run programs. This is the moment the device stops being "just a chip with power" and becomes a working computer.

5. The first program (init)

Linux always starts exactly one program itself. It is traditionally called init (short for "initialise"). Think of it as the foreman: it does not do the real work, but it starts everyone who does and keeps an eye on them.

6. Background programs (services)

The foreman now starts the helpers that run quietly in the background. These are called services — for example the program that manages the network, or the one that answers logins. They stay running the whole time the device is on.

7. The network comes up

Finally the networking services finish their setup and the fiber and Ethernet connections come online. The device is now doing its real job: carrying your Internet.

Why this matters: if the device stops working, the stage where it stops tells you where the problem is. A device that never gets past the bootloader has a very different fault from one that boots fully but has no network.

You can watch it happen

Every stage prints short lines of text as it runs. With a UART connection (Part 4) you see these lines appear live while the device starts. This is called the boot log. Reading it is completely safe — you are only watching, not changing anything.

The exact wording of the log is decided by the people who built the device, so it differs from model to model. But the shape is always the same: bootloader messages first, then Linux messages, then services starting.

Examples

  • A home PC does the same thing: a maker's logo (start code and bootloader), then the operating system loads, then the login screen (services ready).
  • A phone too: the logo, then the animation while the system starts, then the home screen once the background programs are up.
  • Our device has no screen, so instead of a logo and a home screen it shows the same story as text over UART.

Try it yourself (safe)

Below is a sample boot log. It is invented for this lesson — it is not from any real Nokia G-010G-P, and the wording on a real device will differ. Read each line and decide which stage it belongs to.

[ example log — invented for practice ]
U-Boot 2018.03 starting
Loading operating system from storage...
Starting Linux ...
Linux version 4.x  detected 128 MB memory
init: starting first program
service: network manager started
service: login service started
eth0: link is up — network ready

Only ever read a boot log. Do not type into the bootloader or press keys to stop it unless you know exactly what a command does — the bootloader can change or erase storage.

Self-check: match each line to its stage
  • U-Boot ... starting → the bootloader (stage 3).
  • Loading operating system from storage → still the bootloader, fetching Linux.
  • Starting Linux and Linux version ... → the Linux operating system starting (stage 4).
  • init: starting first program → the first program, init (stage 5).
  • service: network manager started and login service started → background services (stage 6).
  • eth0: link is up — network ready → the network coming up (stage 7).

Recap

  • Booting is a fixed sequence of stages, each handing over to the next.
  • The order is: power → built-in start code → bootloader → Linux → first program (init) → services → network.
  • Every stage prints text you can watch live over UART.
  • Where the boot stops tells you where a fault is.

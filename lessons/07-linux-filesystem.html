Part 7 — Understanding the Linux File System

How Linux arranges every file into one big tree of folders.

TL;DR

In Linux, every file and folder hangs off one starting point written as a single slash: /. There are no separate drive letters — everything is one tree.

A handful of folders have fixed jobs: programs, settings, devices, live system info, scratch space, and mount points.

Linux treats almost everything as if it were a file — even hardware. On small devices, the system folders are often locked as read-only so they cannot be changed by accident.

One tree, starting at /

On some systems each disk gets its own letter. Linux does it differently: there is a single tree, and it starts at a spot written as one forward slash, /. This is called the root of the file system — the base that everything else branches off.

Two different "roots" — do not mix them up. The root of the file system is a place: the top folder, /. The root user (from Part 2) is a person: the all-powerful account. Same word, completely different meaning. Here we mean the place.

The folders worth knowing

Each top folder has a job. You do not need to memorise all of them — these are the ones that come up again and again:

FolderWhat lives there
/binPrograms — the everyday commands the system can run.
/etcSettings — plain text configuration files for the system.
/devDevices — the hardware, presented as file-like entries.
/procLive system info — details about what is running right now.
/tmpScratch space — temporary files that can be thrown away.
/mntMount points — where extra storage gets attached to the tree.

What "mount" means

Because there are no drive letters, extra storage has to be joined into the one tree at a chosen folder. Attaching it like that is called mounting, and the folder where it appears (often under /mnt) is the mount point. After mounting, the extra storage just looks like another folder.

"Everything is a file"

Linux has a neat trick: it presents almost everything as if it were a file, so the same simple actions — read from it, write to it — work on very different things. A document is a file, of course. But a piece of hardware in /dev also looks like a file, and the live system details in /proc read like files too.

The payoff: once you know how to read a file, you already know how to peek at hardware and at the running system. One idea, used everywhere.

System folders are often read-only

On small devices, the folders holding Linux and its programs are usually set to read-only: they can be read but not changed while the device runs. This protects the system — a stray command cannot quietly corrupt it — and it means the operating system starts from a known-good state every time. Scratch areas like /tmp stay writable so programs still have somewhere to work.

Read-only system folders are a safety feature, not a fault. If a folder refuses to let you change it, that is often the device protecting itself exactly as intended.

Examples

  • The single tree is like one filing cabinet for the whole building — instead of a separate cabinet per person, there is one, and everything has a labelled drawer inside it.
  • /dev as files is like a light switch on the wall: you do not see the wiring, you just flip a simple control and the hardware responds.
  • Read-only system folders are like a museum display behind glass: you can look all you want, but you cannot rearrange the exhibit by accident.

Try it yourself (safe)

If you have a Linux PC (this course's device is not needed here), open a terminal and type one read-only command:

ls /

ls means "list". This only lists what sits at the root of the file system — it reads, it never changes anything. You should see folder names like bin, etc, dev and others from the table above.

Stick to ls for now. It is a look-only command. Avoid commands that write, move or delete until a later part explains them — and never run commands you do not understand on a device you care about.

No Linux PC? Do the matching exercise instead: /bin, /etc, /dev, /tmp — which one holds programs, which holds settings, which holds hardware, which is scratch space?

Self-check
  • ls / lists the top-level folders and changes nothing — safe to run any time.
  • Matching: /bin → programs; /etc → settings; /dev → hardware (as files); /tmp → scratch space.

Recap

  • Linux has one tree, starting at the root of the file system: /.
  • The root of the file system (a place) is not the root user (a person).
  • Key folders: /bin programs, /etc settings, /dev devices, /proc live info, /tmp scratch, /mnt mount points.
  • Almost everything is presented as a file, so the same simple actions work everywhere.
  • On small devices the system folders are often read-only to protect them.

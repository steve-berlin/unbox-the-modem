Part 14 — Common File Systems Used in Embedded Linux

How small devices arrange their files, and why the important part is kept read-only.

TL;DR

A file system is just a way of organising files on storage so the device can find them.

Small devices usually use a few different ones at once: a locked-down, squeezed one for the operating system, a writable one for your settings, and a memory-only one for scratch data that vanishes when the power goes.

The operating system is kept read-only on purpose — so it can't get scrambled, and the device stays reliable.

What is a file system?

Storage on its own is just a big space to keep data. A file system is the filing method laid over that space: it decides how files are named, where each one lives, and how folders are arranged. Without one, the device would just see a blob and have no way to find anything.

Your PC uses a file system too. Small "embedded" devices like this one are unusual only in that they often use several at the same time, each chosen for a different job.

The read-only, squeezed one (SquashFS)

SquashFS holds the operating system. Two things make it a good fit: it is compressed (squeezed small so it fits in limited storage), and it is read-only (nothing can change it while the device runs). A locked operating system can't be corrupted by a crash or a mistake.

The writable ones (JFFS2 and UBIFS)

Settings have to change — your Wi-Fi name, passwords, ISP details. Those live on a writable file system. JFFS2 and UBIFS are two common ones built specially for the kind of permanent storage these devices use. You don't need to tell them apart; just know they are the writable settings area.

The memory-only ones (tmpfs and initramfs)

Some data is only needed while the device is running — temporary notes, logs in progress, files unpacked during start-up. These sit in tmpfs or initramfs, which live entirely in RAM. Because RAM is wiped when power is removed, everything here disappears on reboot. That's a feature, not a bug: it keeps clutter from piling up.

Stacking a writable layer on a read-only base (overlay)

Here's the clever trick that ties it together. If the operating system is read-only, how can the device ever save a changed file? The answer is an overlay: a thin writable layer laid on top of the locked base.

You see one combined set of files. Reads come from the locked base underneath; any change is written to the thin layer on top, which quietly hides the original. The base is never touched, yet things still appear to change. It's like a sheet of clear plastic over a printed page — you can write on the plastic without altering the page below.

Why keep the operating system read-only?

It comes down to safety and reliability:

  • It can't be corrupted. A power cut or crash while writing can't scramble a part that is never written.
  • It always starts the same way. The base is identical every boot, so behaviour is predictable.
  • Mistakes are contained. A bad change lands on the throwaway layer, not the operating system.
  • It saves space. Being compressed, the locked base fits in the device's small storage.

Examples

  • You change the Wi-Fi name. The new value is written to the JFFS2/UBIFS settings area, so it survives a reboot.
  • The device unpacks some helper files during start-up into tmpfs; after a reboot they're gone and get unpacked fresh again.
  • You edit a system file that lives on the read-only base. Thanks to the overlay, your edit lands on the top layer — but a factory reset that clears that layer brings the original straight back.

Try it yourself (safe)

No device needed — just match each file system to its job. Think about which needs to survive a reboot and which does not.

  1. SquashFS
  2. JFFS2 / UBIFS
  3. tmpfs / initramfs

Jobs: (a) holds your saved settings; (b) holds the locked operating system; (c) scratch space that vanishes on reboot.

Show the answers

1 → b: SquashFS holds the locked, read-only operating system.

2 → a: JFFS2 / UBIFS hold your writable, saved settings.

3 → c: tmpfs / initramfs are memory-only scratch space that disappears on reboot.

Exactly which file systems a given device uses, and how its storage is divided, is something you read off your own unit later — not something to assume. This lesson teaches the general kinds so those readings make sense.

Recap

  • A file system is how files are organised on storage.
  • Small devices use several at once, each for a different job.
  • SquashFS: read-only, compressed, holds the operating system.
  • JFFS2 / UBIFS: writable, hold your settings.
  • tmpfs / initramfs: memory-only, vanish on reboot.
  • An overlay stacks a thin writable layer on the locked base, so things can change without touching the operating system.
  • Keeping the operating system read-only makes the device safer and more reliable.

Part 10 — How Bootloaders Work

The tiny program that runs before Linux and decides what to start.

TL;DR

A bootloader is the first software that runs when the device powers on. Its whole job is to find the operating system and start it.

Many embedded devices use a bootloader called U-Boot.

It often offers a tiny text prompt on the console where you can pause the start-up and type commands. It also keeps a set of settings, and it can help recover a device that won't boot on its own.

Those same powers make it easy to break things — change its settings only when you understand exactly what you are doing.

What the bootloader is for

When you plug the device in, the operating system is not running yet — it is sitting in permanent storage, waiting. Something has to fetch it, put it into memory, and hand control over to it. That something is the bootloader. Think of it as the starter motor in a car: it does not drive anywhere itself, but nothing else can start until it has done its one small job.

Part 2 met this idea in passing. Here we look at it properly.

U-Boot: a common bootloader

There are several bootloaders in the world, but a very common one on small Linux devices is called U-Boot ("Universal Boot Loader"). If you ever watch a device start over a console cable, U-Boot is one of the names you are most likely to see scroll past. You should verify which bootloader your own device uses by reading what it prints as it starts — do not assume.

The tiny text prompt

Many bootloaders, U-Boot included, will pause for a moment early in start-up and offer a small text prompt on the console — often just a short symbol like => waiting for you to type. Usually you have only a second or two, and you interrupt the countdown by pressing a key. If you do nothing, the countdown finishes and the device boots normally.

At that prompt you can type commands: to show settings, to print information about the storage, or to start the operating system manually. It is a separate little world that exists before Linux — Linux is not running yet when you are here.

Settings the bootloader keeps

The bootloader stores a set of its own settings, usually called environment variables — simply named values it remembers, such as where to find the operating system or which options to pass it when starting. These are read every time the device boots. Changing one changes how the device starts next time.

It can also recover a broken device

Because the bootloader runs before the operating system, it still works even when the operating system is damaged and will not start. That makes it a lifeline: from the bootloader you can often load a fresh, working copy of the system back onto the device and bring it back to life. This is exactly why the backups this course keeps insisting on matter — the bootloader gives you a place to write a good copy back from.

Examples

  • The normal case: you power on, the bootloader counts down for a second, nobody presses a key, and it quietly starts the operating system. You never see it.
  • Pausing it: an engineer watching the console presses a key during the countdown, lands at the => prompt, and reads out the current settings before letting boot continue.
  • A rescue: a device's operating system is corrupted and won't start. From the bootloader, a technician loads a fresh working copy back onto it, and the device lives again.

Try it yourself (safe)

Below is a SAMPLE — a made-up example, not from any real device. Read it and work out what the two commands would do. Do not connect to the real device.

Hit any key to stop autoboot:  0
=> printenv
bootcmd=load the operating system, then run it
bootdelay=2
=> help
printenv   - show the stored settings
boot       - start the operating system now

This is a printed example only. Do not attach a console cable to the fiber device or type anything at its real prompt during this course — pausing or changing a live bootloader can leave a device unable to start.

Self-check: what would printenv do, and what would boot do?

printenv shows the bootloader's stored settings (its environment variables), such as which command it uses to start the operating system. boot tells the bootloader to go ahead and start the operating system now, ending the pause. Neither, in this sample, changes any settings — they read and act, but do not rewrite storage.

Recap

  • The bootloader is the first software to run; its job is to find and start the operating system.
  • U-Boot is a bootloader common on small Linux devices — verify which one your device uses by reading what it prints.
  • It often offers a tiny text prompt where you can pause boot and type commands.
  • It keeps its own settings, usually called environment variables.
  • Because it runs before the operating system, it can recover a device that won't otherwise start.
  • Its power cuts both ways — careless changes to its settings can stop a device from booting.

Part 16 — Introduction to Cross-Compiling Linux Software

Building programs on your PC that run on the device.

TL;DR

A program starts life as human-written text called source code. Turning that text into something the machine can actually run is called compiling.

The catch: your PC and the device speak different machine languages. A program built for your PC won't run on the device, and the other way round.

A cross-compiler solves this. It runs on your PC but makes programs for the device. That's the whole idea of this lesson.

What does "compiling" mean?

People write programs in a language they can read, such as C. The machine cannot run that text directly. A tool called a compiler translates the text into the numbers a processor understands. Those numbers are the finished program.

So: source code (text a person wrote) goes in, and a runnable program comes out.

Why can't I just copy my PC program onto the device?

Every processor understands one particular machine language, called its instruction set. This is a fact you can rely on: a program compiled for one instruction set will not run on a different one.

Most PCs and laptops use a family called x86 (sometimes written x86-64 or amd64). Many small devices — modems, routers, cameras — use a different family, often ARM or MIPS. Because the languages differ, a program you build on your x86 PC is meaningless to an ARM or MIPS device.

You cannot tell a device's instruction set just by looking at the case. You read it from the device itself — for example from the processor's markings, or from the operating system already on it. Part 12 covered finding that kind of detail on your own unit.

Host and target

Two words make the rest of this easy:

  • Host — the machine you build on. That's your PC.
  • Target — the machine you build for. That's the device.

Normal compiling is when host and target are the same (build on your PC, run on your PC). Cross-compiling is when they differ: build on the host, run on the target.

A quick word on "toolchain"

You'll see the word toolchain. It just means the compiler plus its helper tools bundled together — the little set of programs needed to turn source code into a working program for one particular target. A "cross-toolchain" is that bundle aimed at a device instead of your PC. You don't need to master it now; just recognise the word when it appears.

Examples

  • You write a small program on your x86 laptop and want it to run on an ARM device. You compile it with an ARM cross-compiler, copy the result to the device, and it runs there.
  • A friend hands you a program that runs fine on their PC. It won't run on the device — same reason a French instruction manual doesn't help someone who only reads Japanese. Different language, no translation.
  • Big projects (covered next lesson) build hundreds of programs for a device this exact way, all cross-compiled on a powerful PC because the device itself is too small and slow to do the building.

Quick quiz

1. What does "compiling" turn source code into?

2. In cross-compiling, what is the "target"?

3. Why won't a program built for an x86 PC run on an ARM device?

Try it yourself (safe)

This only reads information — it changes nothing.

  1. On a Linux or Mac PC, open a terminal and run uname -m. On Windows, open Settings → System → About and look at the system type. Note what your PC reports.
  2. Most PCs will show something in the x86 family (like x86_64 or "64-bit x86"). Write it down.
  3. Now think: your device almost certainly uses a different family, likely ARM or MIPS. You'll confirm the exact one on your own unit later — don't assume it here.

Nothing here touches the device. uname -m and the Windows About screen only report information; they don't write or install anything.

Self-check: your PC says x86_64 and your device is ARM. Can you copy a program straight across?

No. Different instruction sets mean the program is unreadable to the other machine. You'd need to cross-compile the program for ARM first.

Recap

  • Compiling turns human-written source code into a runnable program.
  • A processor understands only its own machine language (instruction set).
  • Your PC (host) is usually x86; the device (target) is often ARM or MIPS.
  • A cross-compiler runs on the host but produces programs for the target.
  • A "toolchain" is just the compiler plus its helper tools.
